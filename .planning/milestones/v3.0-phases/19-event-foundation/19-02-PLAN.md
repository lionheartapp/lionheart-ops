  <name>Task 1: Create Zod schemas/types and EventProject service with activity logging</name>
  <files>src/lib/types/event-project.ts, src/lib/services/eventProjectService.ts</files>
  <action>
    NOTE: Unit tests for service functions are deferred to a future testing phase. Service correctness is validated via import checks, TypeScript compilation, and manual API testing in Plans 04-06.

    1. Create `src/lib/types/event-project.ts`:
       - Define Zod schemas: CreateEventProjectSchema, UpdateEventProjectSchema, CreateScheduleBlockSchema, UpdateScheduleBlockSchema, CreateEventTaskSchema, UpdateEventTaskSchema, CreateEventSeriesSchema, UpdateEventSeriesSchema
       - Export inferred TypeScript types from each schema
       - Title: z.string().min(1).max(200), dates: z.string().datetime() or z.coerce.date()
       - Schedule block: type is z.enum matching EventScheduleBlockType values, title required, startsAt/endsAt required
       - Task: title required, status z.enum matching EventTaskStatus, priority z.enum matching EventTaskPriority, category optional string, assigneeId optional string, dueDate optional

    2. Create `src/lib/services/eventProjectService.ts`:
       - Import `prisma` from `@/lib/db` (org-scoped)
       - `appendActivityLog(eventProjectId, actorId, type, metadata?)` — creates immutable EventActivityLog row. This is a private helper called by every other function.
       - `createEventProject(data, createdById, source, sourceId?)` — Creates EventProject. If source is DIRECT_REQUEST, status is PENDING_APPROVAL. If PLANNING_SUBMISSION or SERIES, status is CONFIRMED (and calls confirmEventProject internally). Always appends CREATED activity log.
       - `getEventProject(id)` — findUnique with includes for scheduleBlocks, tasks, activityLog (ordered by createdAt desc), createdBy, approvedBy.
       - `listEventProjects(filters?)` — findMany with optional status filter, ordered by startsAt.
       - `updateEventProject(id, data, actorId)` — Updates fields. Compares old vs new to build metadata for FIELD_UPDATED activity log entry. Returns updated project.
       - `approveEventProject(id, approverId)` — Validates current status is PENDING_APPROVAL. Updates to CONFIRMED, sets approvedById and approvedAt. Appends APPROVAL_GRANTED log. Then calls internal confirmEventProject to create CalendarEvent bridge.
       - `rejectEventProject(id, actorId, reason?)` — Updates to CANCELLED with APPROVAL_REJECTED log.
       - `confirmEventProject(id, actorId)` — Creates CalendarEvent bridge with `sourceModule: 'event-project'`, `sourceId: id`. Uses the project's calendarId (or org default calendar). Appends STATUS_CHANGE log.
       - `createScheduleBlock(eventProjectId, data, actorId)` — Creates block, appends SCHEDULE_BLOCK_ADDED log.
       - `updateScheduleBlock(blockId, data, actorId)` — Updates block, appends SCHEDULE_BLOCK_UPDATED log.
       - `deleteScheduleBlock(blockId, actorId, eventProjectId)` — Deletes block (hard delete, schedule blocks are not soft-deleted), appends SCHEDULE_BLOCK_REMOVED log.
       - `createEventTask(eventProjectId, data, actorId)` — Creates task, appends TASK_CREATED log.
       - `updateEventTask(taskId, data, actorId, eventProjectId)` — Updates task. If status changes to DONE, sets completedAt and appends TASK_COMPLETED. Otherwise appends TASK_UPDATED.
       - `getActivityLog(eventProjectId)` — Returns activity log entries ordered by createdAt desc, with actor included.

    IMPORTANT:
    - Use `prisma` (org-scoped) for all operations — these run inside runWithOrgContext in route handlers.
    - Never update or delete EventActivityLog rows.
    - For confirmEventProject, if calendarId is null on the project, query for the org's default calendar and use that. If no calendar exists, skip bridge creation (log a warning).
    - Follow immutable patterns: return new objects from updates, do not mutate inputs.
  </action>
