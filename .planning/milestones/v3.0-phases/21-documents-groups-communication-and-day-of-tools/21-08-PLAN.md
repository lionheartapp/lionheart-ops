**src/lib/services/eventCheckInService.ts:**

- `checkIn(data: { eventProjectId, registrationId, checkedInById?, method?: 'QR_SCAN' | 'MANUAL', syncedAt?: Date })` — creates an EventCheckIn record. Uses upsert with unique constraint [registrationId, eventProjectId] — if already checked in, return the existing record with an `alreadyCheckedIn: true` flag (do not error). If new check-in, set checkedInAt to now(). If syncedAt is null, it was created offline (will be synced later). Returns the check-in record plus participant info.

- `undoCheckIn(eventProjectId: string, registrationId: string)` — deletes the check-in record (for mistakes — staff accidentally scanned wrong person).

- `getCheckInCounter(eventProjectId: string)` — returns { checkedIn: number, total: number } where total is count of REGISTERED participants and checkedIn is count of EventCheckIn records for this event.

- `getCheckInStatus(eventProjectId: string)` — returns full list: all REGISTERED participants with isCheckedIn boolean and checkedInAt timestamp. Sorted by last check-in time (most recent first), with unchecked-in at the bottom. Each entry includes: registrationId, firstName, lastName, photoUrl, grade, isCheckedIn, checkedInAt, checkedInBy (name).

- `getParticipantByRegistration(registrationId: string)` — fetches comprehensive participant info for QR scan result display. Returns:
  - Basic: firstName, lastName, photoUrl, grade, email.
  - Medical flags (only if caller has medical permission — accept a boolean param `includeMedical`): allergies, medications (from RegistrationSensitiveData).
  - Groups: list of group assignments with group name and type.
  - Schedule: list of EventScheduleBlocks for the event.
  - Announcements: list of relevant announcements (using getAnnouncementsForRegistration if available, or simple query).
  - Check-in status: isCheckedIn, checkedInAt.

- `syncOfflineCheckIns(checkIns: { registrationId, eventProjectId, checkedInAt, method }[])` — batch upsert for offline sync. Skips duplicates. Returns { synced: number, skipped: number }.

**src/lib/services/eventIncidentService.ts:**

- `createIncident(data: { eventProjectId, type: IncidentType, severity: IncidentSeverity, description, actionsTaken?, followUpNeeded?, followUpNotes?, photoUrl?, reportedById, participantIds?: string[], syncedAt?: Date })` — creates EventIncident and EventIncidentParticipant junction records. Returns the incident with participants.

- `updateIncident(id: string, data: Partial<{ description, actionsTaken, followUpNeeded, followUpNotes, severity, photoUrl }>)` — update incident details.

- `listIncidents(eventProjectId: string, options?: { type?: IncidentType, severity?: IncidentSeverity })` — returns incidents for the event with involved participant names, sorted by createdAt desc. Optional filters by type and severity.

- `getIncident(id: string)` — returns single incident with full participant details.

- `deleteIncident(id: string)` — hard-delete.

- `syncOfflineIncidents(incidents: Array<Omit<CreateIncidentInput, 'syncedAt'>>)` — batch create for offline sync. Sets syncedAt to now(). Returns { synced: number }.

**1. `/api/events/projects/[id]/check-in/route.ts`:**
- GET: assertCan(EVENTS_CHECKIN_MANAGE). Returns getCheckInCounter(id) and optionally getCheckInStatus(id) if `?full=true` query param.
- POST: assertCan(EVENTS_CHECKIN_MANAGE). Zod validates { registrationId, method? }. Calls checkIn with ctx.userId as checkedInById. Returns check-in record + participant flash card data.
- PUT: Sync endpoint. assertCan(EVENTS_CHECKIN_MANAGE). Zod validates { checkIns: array of { registrationId, checkedInAt, method } }. Calls syncOfflineCheckIns. Returns { synced, skipped }.

**2. `/api/events/projects/[id]/check-in/[registrationId]/route.ts`:**
- GET: assertCan(EVENTS_CHECKIN_MANAGE). Returns full participant info via getParticipantByRegistration(registrationId, includeMedical: can(userId, EVENTS_MEDICAL_READ)). Medical data included only if caller has medical permission.
- DELETE: assertCan(EVENTS_CHECKIN_MANAGE). Calls undoCheckIn. Returns 200.

**3. `/api/events/projects/[id]/incidents/route.ts`:**
- GET: assertCan(EVENTS_INCIDENTS_MANAGE). Optional query params: `?type=MEDICAL&severity=SERIOUS`. Returns listIncidents(id, filters).
- POST: assertCan(EVENTS_INCIDENTS_MANAGE). Zod validates { type, severity, description, actionsTaken?, followUpNeeded?, followUpNotes?, photoUrl?, participantIds? }. Calls createIncident with ctx.userId as reportedById. Returns 201.
- PUT: Sync endpoint. assertCan(EVENTS_INCIDENTS_MANAGE). Zod validates { incidents: array }. Calls syncOfflineIncidents. Returns { synced }.

**4. `/api/events/projects/[id]/incidents/[incidentId]/route.ts`:**
- GET: assertCan(EVENTS_INCIDENTS_MANAGE). Returns getIncident(incidentId).
- PUT: assertCan(EVENTS_INCIDENTS_MANAGE). Zod validates update fields. Calls updateIncident.
- DELETE: assertCan(EVENTS_INCIDENTS_MANAGE). Calls deleteIncident.

**5. `/api/events/check-in/[registrationId]/route.ts` (PUBLIC — participant self-service, QR-03):**
This is the public endpoint that a participant accesses when they scan their QR code. No auth required — the registration ID in the QR code serves as the access token (same pattern as portal magic links).
- GET: Fetches registration by ID using rawPrisma. If not found or deleted, return 404. Returns:
  - Participant name, photo.
  - Event info: title, dates, location.
  - Schedule blocks.
  - Group assignments (name, type).
  - Announcements (filtered for this participant).
  - Check-in status.
  - Does NOT include medical data (FERPA — never expose via public endpoint).
- Add `/api/events/check-in/` to middleware public paths if not already covered.
