<task type="auto">
  <name>Task 1: Create group and activity service layer</name>
  <files>src/lib/services/eventGroupService.ts</files>
  <action>
Create `src/lib/services/eventGroupService.ts` with:

**Group CRUD:**
- `createGroup(data: { eventProjectId, name, type: EventGroupType, capacity?, leaderId?, description?, sortOrder? })` — creates an EventGroup. Validates type is valid enum.
- `updateGroup(id: string, data: Partial<...>)` — update name, type, capacity, leader, description, sortOrder.
- `deleteGroup(id: string)` — hard-delete (cascade deletes assignments).
- `listGroups(eventProjectId: string, type?: EventGroupType)` — returns groups filtered by optional type, with assignment count and leader info (name, avatar). Sorted by sortOrder.

**Assignments:**
- `assignToGroup(registrationId: string, groupId: string, assignedById?: string)` — creates EventGroupAssignment. Check capacity: if group has capacity set and assignment count >= capacity, throw error "Group is at capacity".
- `removeFromGroup(registrationId: string, groupId: string)` — deletes the assignment.
- `getGroupAssignments(groupId: string)` — returns all assignments for a group with participant info (name, grade, photoUrl, medical flags).
- `getUnassignedParticipants(eventProjectId: string, groupType: EventGroupType)` — returns REGISTERED participants not assigned to any group of the specified type. Important: a participant can be in a BUS group AND a CABIN group — these are independent. Query: find registrations where registrationId NOT IN (select registrationId from EventGroupAssignment where groupId IN (select id from EventGroup where eventProjectId=X and type=Y)).

**Auto-Assign:**
- `autoAssign(eventProjectId: string, groupType: EventGroupType, options?: { balanceBy?: 'grade' | 'gender' })` — distributes unassigned participants into groups of the specified type. Algorithm:
  1. Get all groups of this type with current counts.
  2. Get unassigned participants for this type.
  3. Sort groups by available capacity (most space first).
  4. If balanceBy is specified, sort participants by the balance field, then round-robin into groups.
  5. Otherwise, distribute evenly (round-robin).
  6. Skip groups at capacity. Throw error if all groups full but participants remain.
  7. Returns count of assignments made.

**Activities:**
- `createActivity(data: { eventProjectId, name, description?, capacity?, scheduledAt?, durationMinutes?, locationText?, sortOrder? })` — creates EventActivityOption.
- `updateActivity(id: string, data: Partial<...>)` — update fields.
- `deleteActivity(id: string)` — hard-delete (cascade deletes signups).
- `listActivities(eventProjectId: string)` — returns activities with signup count vs capacity.

**Activity Signups:**
- `signupForActivity(registrationId: string, activityId: string)` — creates EventActivitySignup. Check capacity: if count >= capacity, throw error "Activity is full".
- `cancelActivitySignup(registrationId: string, activityId: string)` — deletes signup.
- `getActivitySignups(activityId: string)` — returns all signups with participant info.

**Dietary/Medical Report:**
- `getDietaryMedicalReport(eventProjectId: string)` — aggregates dietary needs and medical info across all REGISTERED participants. Requires events:medical:read permission (caller must check before calling). Queries RegistrationSensitiveData through registrations. Returns:
  - dietarySummary: { need: string, count: number }[] (aggregated from registration.dietaryNeeds)
  - allergySummary: { allergy: string, count: number, participants: { name, registrationId }[] }[]
  - medicationCount: number (count of participants with medications)
  - participantsWithMedicalNotes: { name, registrationId }[] (just names, not the notes themselves — notes require individual lookup)

Create 6 API route files following the standard route pattern.

**1. `/api/events/projects/[id]/groups/route.ts`:**
- GET: assertCan(EVENTS_GROUPS_MANAGE). Optional `?type=BUS` filter. Returns listGroups(id, type?).
- POST: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { name, type, capacity?, leaderId?, description? }. Returns 201.
- PUT: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { groupId, ...updateFields }. Returns updated group.
- DELETE: assertCan(EVENTS_GROUPS_MANAGE). Requires { groupId }. Returns 200.

**2. `/api/events/projects/[id]/groups/[groupId]/assignments/route.ts`:**
- GET: assertCan(EVENTS_GROUPS_MANAGE). Returns getGroupAssignments(groupId). Also includes unassigned participants for this group's type via getUnassignedParticipants.
- POST: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { registrationId }. Calls assignToGroup with ctx.userId as assignedById. Handles capacity error as 409.
- DELETE: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { registrationId }. Calls removeFromGroup.

**3. `/api/events/projects/[id]/groups/auto-assign/route.ts`:**
- POST: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { groupType, balanceBy? }. Calls autoAssign. Returns { assignmentsCreated: number }.

**4. `/api/events/projects/[id]/activities/route.ts`:**
- GET: assertCan(EVENTS_GROUPS_MANAGE). Returns listActivities(id).
- POST: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { name, capacity?, scheduledAt?, durationMinutes?, locationText? }. Returns 201.
- PUT: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { activityId, ...fields }. Returns updated.
- DELETE: assertCan(EVENTS_GROUPS_MANAGE). Requires { activityId }.

**5. `/api/events/projects/[id]/activities/signups/route.ts`:**
- GET: assertCan(EVENTS_GROUPS_MANAGE). Requires `?activityId=`. Returns getActivitySignups.
- POST: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { registrationId, activityId }. Handles capacity error as 409.
- DELETE: assertCan(EVENTS_GROUPS_MANAGE). Zod validates { registrationId, activityId }.

**6. `/api/events/projects/[id]/dietary-medical/route.ts`:**
- GET: assertCan(EVENTS_MEDICAL_READ) — note this uses the FERPA-gated medical permission, NOT the groups permission. Returns getDietaryMedicalReport(id).
