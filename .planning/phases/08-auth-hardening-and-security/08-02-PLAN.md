<task type="auto">
  <name>Task 1: Rate limiter utility module</name>
  <files>src/lib/rate-limit.ts</files>
  <action>
Create `src/lib/rate-limit.ts` implementing an in-memory sliding window rate limiter.

**RateLimiter class:**
- Constructor accepts `{ windowMs: number, maxAttempts: number }` (e.g., `{ windowMs: 15 * 60 * 1000, maxAttempts: 5 }`)
- Internal `Map<string, { timestamps: number[] }>` keyed by IP string
- `check(key: string): { allowed: boolean; remaining: number; retryAfterMs: number }` — slides the window, counts attempts, returns status
- `increment(key: string): void` — records an attempt timestamp
- `reset(key: string): void` — clears all attempts for key (used on successful login)
- Sliding window: filter out timestamps older than `windowMs` before counting
- Cleanup interval: every 5 minutes, prune entries with no timestamps in the window to prevent memory leaks. Use `setInterval` — store the interval ref for potential cleanup.

**Pre-configured instances (exported):**
- `loginRateLimiter = new RateLimiter({ windowMs: 15 * 60 * 1000, maxAttempts: 5 })` — login: 5 per 15min
- `publicApiRateLimiter = new RateLimiter({ windowMs: 60 * 1000, maxAttempts: 30 })` — general public: 30 per minute
- `signupRateLimiter = new RateLimiter({ windowMs: 60 * 60 * 1000, maxAttempts: 5 })` — signup: 5 per hour

**Also export:**
- `getRateLimitHeaders(result: { remaining: number; retryAfterMs: number }): Record<string, string>` — returns `{ 'X-RateLimit-Remaining': string, 'Retry-After': string }` for response headers

**Important:** This is a single-process in-memory solution suitable for Vercel's serverless functions (each function instance has its own limiter). For horizontal scaling, Redis would be needed (deferred to v2.1 per user decision). In practice, Vercel cold starts mean this provides best-effort protection — sufficient for launch.

  <name>Task 2: Wire rate limiting into login route and middleware</name>
  <files>src/app/api/auth/login/route.ts, src/middleware.ts</files>
  <action>
1. **Login route** (`src/app/api/auth/login/route.ts`):
   - Import `loginRateLimiter` and `getRateLimitHeaders` from `@/lib/rate-limit`
   - Import `getIp` from `@/lib/services/auditService`
   - At the TOP of the POST handler (before any body parsing): extract IP via `getIp(req)`, use `'unknown'` as fallback
   - Call `loginRateLimiter.increment(ip)` then `loginRateLimiter.check(ip)`
   - If `!allowed`: return `NextResponse.json(fail('RATE_LIMITED', 'Too many login attempts. Please try again later.'), { status: 429, headers: getRateLimitHeaders(result) })` — note: do NOT reveal "in X minutes" to avoid information leakage about window size
   - On successful login (after valid credentials confirmed): call `loginRateLimiter.reset(ip)` to reward correct credentials
   - Keep all existing logic untouched (org context, user lookup, bcrypt compare, JWT signing, audit log)

2. **Middleware** (`src/middleware.ts`):
   - Import `publicApiRateLimiter`, `signupRateLimiter`, `getRateLimitHeaders` from `@/lib/rate-limit`
   - In the org-scoped routes section, BEFORE the `isPublicPath` check, add rate limiting for specific public API paths:
     - If pathname is `/api/organizations/signup`: use `signupRateLimiter`
     - If pathname starts with `/api/auth/forgot-password` or `/api/auth/set-password`: use `publicApiRateLimiter`
   - Extract IP from `req.headers.get('x-forwarded-for')?.split(',')[0]?.trim() || req.headers.get('x-real-ip') || 'unknown'`
   - Call `limiter.increment(ip)` then `limiter.check(ip)`. If not allowed, return `NextResponse.json({ ok: false, error: { code: 'RATE_LIMITED', message: 'Too many requests. Please try again later.' } }, { status: 429, headers: getRateLimitHeaders(result) })`
   - Place rate limit check BEFORE the existing public path handling so it runs first
   - Do NOT rate-limit non-API public paths (static assets, pages)
   - Do NOT rate-limit the login endpoint here (it has its own finer-grained limiter in the route handler)
