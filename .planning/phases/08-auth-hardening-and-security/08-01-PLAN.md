  <files>prisma/schema.prisma, src/lib/auth/password-setup.ts, src/lib/email/templates.ts, src/lib/services/emailService.ts, src/app/api/auth/forgot-password/route.ts, src/app/api/auth/reset-password/route.ts</files>
  <action>
1. **Schema**: Add `type String @default("setup")` field to `PasswordSetupToken` model. Run `npm run db:push` to apply.

2. **password-setup.ts**: Add a `getResetLink(token: string): string` function that returns `${appUrl}/reset-password?token=${encodeURIComponent(token)}`. Keep existing `getSetupLink` unchanged.

3. **Email template**: In `src/lib/email/templates.ts`, add a `password_reset` template entry using the existing `wrapLayout()` pattern and brand tokens. Subject: "Reset your password". Content: "Hi {{firstName}}, we received a request to reset your password for {{orgName}}. Click the button below to set a new password. This link expires in 1 hour." CTA button: "Reset Password" linking to `{{resetLink}}`. Blue band: "If you didn't request this, you can safely ignore this email."

4. **emailService.ts**: Add `sendPasswordResetEmail({ to, firstName, orgName, resetLink })` function that calls `sendBrandedEmail('password_reset', to, { firstName, orgName, resetLink, appUrl })`.

5. **POST /api/auth/forgot-password**: Public endpoint (add to `isPublicPath` in middleware — but NOTE: middleware.ts is owned by plan 02, so instead just add the path `/api/auth/forgot-password` which is already covered by the existing `if (pathname.startsWith('/api/auth/')) return true` rule in middleware).
   - Accept `{ email, organizationId }` in body
   - Validate with Zod
   - Always return `{ ok: true, data: { message: "If an account exists, we sent a reset link." } }` (200) — prevents email enumeration
   - Behind the scenes: look up user by email+orgId using `rawPrisma`. If not found or status !== 'ACTIVE', silently return success
   - If user found: invalidate all previous unused reset tokens for that user (`rawPrisma.passwordSetupToken.updateMany({ where: { userId, type: 'reset', usedAt: null }, data: { usedAt: new Date() } })`)
   - Generate token via `generateSetupToken()`, hash via `hashSetupToken()`, store in PasswordSetupToken with `type: 'reset'`, `expiresAt: now + 1 hour`
   - Build reset link via `getResetLink(token)` and send via `sendPasswordResetEmail()`
   - Fire-and-forget audit log: `audit({ organizationId, userId: user.id, action: 'user.password-reset-request', ... })`

6. **POST /api/auth/reset-password**: Public endpoint (already covered by `/api/auth/` prefix in middleware).
   - Accept `{ token, password }` in body
   - Validate token: hash it, find in DB with `type: 'reset'`, check not used, check not expired
   - Validate password length >= 8 (basic check here; full complexity rules come in plan 05)
   - Hash new password with bcryptjs, update user, mark token as used — use `rawPrisma.$transaction([])`
   - After successful reset: call `signAuthToken()` to generate JWT
   - Return `{ ok: true, data: { token, organizationId, user: { id, email, name, avatar, role, ... } } }` — same shape as login response for client compatibility
   - The token returned in JSON body enables auto-login on the client (plan 03 will add cookie setting later)
