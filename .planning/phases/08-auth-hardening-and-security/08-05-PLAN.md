Implement email verification on signup and password complexity rules across all password-setting flows.

Output: User.emailVerified field, verification email + landing page, login gate for unverified users, shared password validation module, reusable PasswordInput component with inline rule indicators.
</objective>

2. **Password complexity** (`src/lib/validation/password.ts` — new file):
   - Define rules as a constant array for both server and client use:
     ```typescript
     export const PASSWORD_RULES = [
       { id: 'length', label: 'At least 8 characters', test: (pw: string) => pw.length >= 8 },
       { id: 'uppercase', label: 'One uppercase letter', test: (pw: string) => /[A-Z]/.test(pw) },
       { id: 'number', label: 'One number', test: (pw: string) => /[0-9]/.test(pw) },
       { id: 'special', label: 'One special character', test: (pw: string) => /[!@#$%^&*()_+\-=\[\]{};':"\\|,.<>\/?`~]/.test(pw) },
     ] as const
     ```
   - `validatePassword(password: string): { valid: boolean; results: Array<{ id: string; label: string; passed: boolean }> }` — runs all rules, returns per-rule results
   - `passwordSchema` — Zod schema: `z.string().min(8).refine(pw => PASSWORD_RULES.every(r => r.test(pw)), { message: 'Password must contain at least 8 characters, one uppercase letter, one number, and one special character' })`
   - Export all

3. **password-setup.ts**: Add `getVerificationLink(token: string): string` that returns `${appUrl}/verify-email?token=${encodeURIComponent(token)}`

4. **Email template**: In `src/lib/email/templates.ts`, add an `email_verification` template. Subject: "Verify your email address". Content: "Hi {{firstName}}, welcome to {{orgName}}! Professional warm tone matching existing templates.

5. **emailService.ts**: Add `sendVerificationEmail({ to, firstName, orgName, verificationLink })` function using `sendBrandedEmail('email_verification', ...)`.

6. **GET /api/auth/verify-email** (new file):
   - Public endpoint (add to isPublicPath or use the existing `/api/auth/` prefix match)
   - Read `token` from URL search params: `req.nextUrl.searchParams.get('token')`
   - Hash token, find in DB with `type: 'email-verification'`, check not used, check not expired (24h)
   - On valid token:
     - Update user: `emailVerified: true`
     - If user status is PENDING, set to ACTIVE (verification completes the signup flow)
     - Mark token as used
     - Sign JWT and set httpOnly cookie (same pattern as login route from plan 03)
     - Also set CSRF cookie (same as login)
     - Redirect to `/dashboard` with 302 (since this is a GET from an email link, redirect is the UX)
   - On invalid/expired token: redirect to `/verify-email?error=invalid` or `/verify-email?error=expired`
   - Use `rawPrisma` since this is a public endpoint (no org context yet)

7. **POST /api/auth/resend-verification** (new file):
   - Accept `{ email, organizationId }` in body
   - Rate limit: use a limiter of 3 per hour per email (import from rate-limit module or inline a simple check using PasswordSetupToken creation timestamps)
   - Look up user by email + orgId. If not found or already verified, return generic success (prevent enumeration)
   - Invalidate previous unused verification tokens for this user
   - Generate new token, store with `type: 'email-verification'`, `expiresAt: now + 24 hours`
   - Send verification email
   - Return `ok({ message: 'If your account exists, a verification email has been sent.' })`
