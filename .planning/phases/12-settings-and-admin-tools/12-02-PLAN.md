Create 4 API routes for billing management. All routes use the standard pattern: `getOrgIdFromRequest` + `getUserContext` + `assertCan(ctx.userId, PERMISSIONS.SETTINGS_BILLING)` + `runWithOrgContext`.

**Stripe initialization helper** (put at top of each route or in a shared util):
```typescript
import Stripe from 'stripe'
function getStripe(): Stripe | null {
  const key = process.env.STRIPE_SECRET_KEY
  if (!key) return null
  return new Stripe(key, { apiVersion: '2024-04-10' })
}
```

If the env var is missing, return a graceful error (not a crash) — billing features are optional until Stripe is configured.

**1. `GET /api/settings/billing/route.ts`** — Returns current subscription info + available plans:
```typescript
// Fetch org's active subscription with plan details
const subscription = await rawPrisma.subscription.findFirst({
  where: { organizationId: orgId, status: { not: 'CANCELED' } },
  include: { plan: true },
  orderBy: { createdAt: 'desc' },
})

// Fetch all active plans for plan comparison
const plans = await rawPrisma.subscriptionPlan.findMany({
  where: { isActive: true },
  orderBy: { displayOrder: 'asc' },
})

return ok({ subscription, plans })
```
Response shape: `{ ok: true, data: { subscription: Subscription & { plan: SubscriptionPlan } | null, plans: SubscriptionPlan[] } }`

**2. `POST /api/settings/billing/change-plan/route.ts`** — Change subscription plan:
- Body: `{ planId: string, preview?: boolean }`
- If `preview=true`: Use Stripe's upcoming invoice API to calculate proration and return preview (amount, description) without making changes
- If `preview=false` (or omitted): Update the Stripe subscription to the new plan's `stripePriceId` with `proration_behavior: 'create_prorations'`. Update local `Subscription.planId`.
- If no `stripeSubscriptionId` exists (free/trial user upgrading), create a new Stripe subscription via `stripe.subscriptions.create()`. Requires `stripeCustomerId` on the org — if missing, create the customer first via `stripe.customers.create({ email, name })`.
- Return updated subscription data.
- Wrap Stripe calls in try/catch and return meaningful error messages.

**3. `POST /api/settings/billing/portal/route.ts`** — Create Stripe Customer Portal session:
- Requires `stripeCustomerId` on the org's subscription. If `stripeCustomerId` is null or missing, return `fail('BAD_REQUEST', 'No billing account configured. Please select a plan first to set up billing.')` with `{ status: 400 }`.
- Call `stripe.billingPortal.sessions.create({ customer: stripeCustomerId, return_url: '...' })` where return_url points back to `/settings` (read from `req.headers.get('origin')` or `process.env.NEXT_PUBLIC_APP_URL`).
- Return `{ url: session.url }` — the client will `window.open()` to this URL.

**4. `GET /api/settings/billing/invoices/route.ts`** — Fetch recent invoices:
- If Stripe is configured and org has `stripeCustomerId`:
  - Fetch from Stripe: `stripe.invoices.list({ customer: stripeCustomerId, limit: 12 })`
  - Map to: `{ id, date, amount, status, pdfUrl, description }`
- Fallback: fetch from local Payment table: `rawPrisma.payment.findMany({ where: { organizationId: orgId }, orderBy: { createdAt: 'desc' }, take: 12 })`
- Return `ok({ invoices: [...] })`

Use `rawPrisma` for subscription/plan queries since these reference organization-level data. The Subscription model is not in the org-scoped extension.
