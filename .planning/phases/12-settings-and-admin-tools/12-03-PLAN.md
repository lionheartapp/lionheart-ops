**1. Add NotificationPreference model to `prisma/schema.prisma`:**

```prisma
model NotificationPreference {
  id             String   @id @default(cuid())
  organizationId String
  userId         String
  type           String   // NotificationType string (e.g. "event_updated")
  emailEnabled   Boolean  @default(true)
  inAppEnabled   Boolean  @default(true)
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  organization Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  user         User         @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, type])
  @@index([userId])
  @@index([organizationId])
}
```

Add the relation to Organization model: `notificationPreferences NotificationPreference[]`
Add the relation to User model: `notificationPreferences NotificationPreference[]`

Also add a `pauseAllNotifications` Boolean field to the User model: `pauseAllNotifications Boolean @default(false)` — this is the master pause toggle.

Run `npm run db:push` to push schema changes.

**2. Create `src/app/api/settings/organization/route.ts`:**

`GET` — Return current org name and slug:
- Auth: `getOrgIdFromRequest` + `getUserContext` + `assertCan(ctx.userId, PERMISSIONS.SETTINGS_READ)`
- Query: `rawPrisma.organization.findUnique({ where: { id: orgId }, select: { name: true, slug: true } })`
- Return `ok({ name, slug })`

`PATCH` — Update org name and/or slug:
- Auth: same + `assertCan(ctx.userId, PERMISSIONS.SETTINGS_READ)` (admin-only via SETTINGS_READ which is admin/super-admin)
- Body: `{ name?: string, slug?: string }`
- Validate with Zod: name min 2 chars, slug lowercase alphanumeric + hyphens, min 3 chars, max 50 chars
- If slug is changing:
  - Check uniqueness: call `rawPrisma.organization.findFirst({ where: { slug: newSlug, id: { not: orgId } } })`
  - If taken, return `fail('VALIDATION_ERROR', 'Slug already taken')`
- Update: `rawPrisma.organization.update({ where: { id: orgId }, data: { name, slug } })`
- Audit: `audit({ organizationId: orgId, userId: ctx.userId, userEmail: ctx.email, action: 'organization.update', resourceType: 'Organization', resourceId: orgId, changes: { name, slug } })`
- Return `ok({ name: updated.name, slug: updated.slug })`

**3. Create `src/app/api/user/notification-preferences/route.ts`:**

`GET` — Return all notification preferences for the current user:
- Auth: `getUserContext(req)` — no special permission needed (users manage their own preferences)
- Query: `rawPrisma.notificationPreference.findMany({ where: { userId: ctx.userId } })`
- Also fetch `rawPrisma.user.findUnique({ where: { id: ctx.userId }, select: { pauseAllNotifications: true } })`
- Return `ok({ preferences: [...], pauseAll: user.pauseAllNotifications })`

`PUT` — Upsert notification preferences:
- Body: `{ preferences: Array<{ type: string, emailEnabled: boolean, inAppEnabled: boolean }>, pauseAll?: boolean }`
- Validate types against known NotificationType values
- For each preference: `rawPrisma.notificationPreference.upsert({ where: { userId_type: { userId: ctx.userId, type } }, create: { organizationId: orgId, userId: ctx.userId, type, emailEnabled, inAppEnabled }, update: { emailEnabled, inAppEnabled } })`
- If `pauseAll` is provided: `rawPrisma.user.update({ where: { id: ctx.userId }, data: { pauseAllNotifications: pauseAll } })` (use rawPrisma to bypass org-scope since it's the user's own record)
- Return `ok({ updated: true })`

**4. Update `src/lib/services/notificationService.ts`:**

In the `createNotification` function, add a preference check BEFORE creating the notification:
```typescript
// Check user's notification preference
const pref = await prisma.notificationPreference.findUnique({
  where: { userId_type: { userId: data.userId, type: data.type } }
})

// Check master pause
const user = await rawPrisma.user.findUnique({
  where: { id: data.userId },
  select: { pauseAllNotifications: true }
})

// If user has paused all notifications, skip both email and in-app
if (user?.pauseAllNotifications) return

// If preference exists and in-app is disabled, skip creating the notification
if (pref && !pref.inAppEnabled) return
```

This ensures notifications respect user preferences. Email preference checking should be done at the point where emails are sent (if that codepath exists in notification dispatch). For now, the in-app check is the critical path.
