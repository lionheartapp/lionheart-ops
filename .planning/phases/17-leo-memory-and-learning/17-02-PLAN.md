<task type="auto">
  <name>Task 1: Modify chat route to persist conversations and messages</name>
  <files>src/app/api/ai/assistant/chat/route.ts</files>
  <action>
Modify the existing `/api/ai/assistant/chat` POST handler to persist messages to the database. The streaming UX must remain unchanged — persistence is a side effect, not a blocking operation.

**Changes to ChatRequestSchema:**
- Add `conversationId: z.string().optional()` to the Zod schema

**At the start of the handler (after validation, inside try block):**
1. If `body.conversationId` is provided, verify it belongs to the current user (getConversation with orgId check). If not found, ignore the ID and create new.
2. If no conversationId, create a new conversation: `createConversation(ctx.userId, orgId)`
3. Save the user's message immediately: `addMessage(conversationId, { role: 'user', content: body.message, organizationId: orgId })`
4. Emit a `conversation_id` SSE event right after stream starts so the frontend gets the ID: `write({ type: 'conversation_id', conversationId })`

**After the final text is computed (after extractMarkers, before the 'done' event):**
1. Save the assistant's response: `addMessage(conversationId, { role: 'assistant', content: finalText, organizationId: orgId })` — capture the returned messageId
2. Include the messageId in the updated conversationHistory's last assistant turn

**Auto-title generation:**
- If this is a NEW conversation (no conversationId in request), after saving the user message, call `updateConversationTitle(conversationId, body.message.slice(0, 100))` — use the first 100 chars of the first user message as the title

**Tool call persistence:**
- When a tool is executed, save tool_call and tool_result messages to the conversation:
  - `addMessage(conversationId, { role: 'tool_call', content: JSON.stringify({ name: fc.name, args: fc.args }), toolName: fc.name, organizationId: orgId })`
  - After execution: `addMessage(conversationId, { role: 'tool_result', content: toolResult, toolName: fc.name, toolSuccess: !parsed.error, organizationId: orgId })`

**Important:** All persistence calls should be fire-and-forget (don't await them in the streaming hot path). Use `void addMessage(...)` pattern so persistence errors don't break the streaming response. Wrap each in its own try/catch that logs but doesn't throw.

**Updated 'done' event:**
- Add `conversationId` to the done event payload (update the StreamEvent type if not already done in Plan 01)

**Do NOT change:** The Gemini streaming loop, SSE format, tool execution logic, or any existing behavior. Only ADD persistence as a side effect.

  <files>src/app/api/conversations/route.ts, src/app/api/conversations/[id]/route.ts, src/app/api/conversations/[id]/messages/route.ts, src/app/api/conversations/[id]/feedback/route.ts</files>
  <action>
Create 4 new API route files following the standard route pattern (getOrgIdFromRequest, getUserContext, runWithOrgContext, ok/fail envelope, Pino logger, Sentry).

**GET /api/conversations** (`src/app/api/conversations/route.ts`):
- Auth required (getUserContext)
- Query params: `limit` (default 20, max 50), `offset` (default 0) — parse with Zod
- Call `getConversations(ctx.userId, orgId, { limit, offset })`
- Return `ok({ conversations, total })` — include total count for pagination
- Add Pino logger: `logger.child({ route: '/api/conversations', method: 'GET' })`

**GET /api/conversations/[id]** (`src/app/api/conversations/[id]/route.ts`):
- Auth required
- Get conversation by ID, verify it belongs to current user's org
- Return `ok(conversation)` or `fail('NOT_FOUND', ...)` with 404
- Add Pino + Sentry instrumentation

**DELETE /api/conversations/[id]** (same file):
- Auth required
- Soft-delete via `deleteConversation(id, orgId)`
- Verify conversation belongs to current user (not just org — users can only delete their own conversations)
- Return `ok({ deleted: true })`

**GET /api/conversations/[id]/messages** (`src/app/api/conversations/[id]/messages/route.ts`):
- Auth required
- Verify conversation belongs to current user's org
- Query params: `limit` (default 100), `offset` (default 0)
- Call `getMessages(id, { limit, offset })`
- Return `ok({ messages })`
- Add Pino + Sentry instrumentation

**POST /api/conversations/[id]/feedback** (`src/app/api/conversations/[id]/feedback/route.ts`):
- Auth required
- Body schema: `{ messageId: string, score: z.number().int().min(1).max(5) }`
- Verify the message belongs to a conversation in this user's org
- Call `setMessageFeedback(messageId, score)`
- Return `ok({ updated: true })`
- Add Pino + Sentry instrumentation

All routes must follow the standard error handling pattern from CLAUDE.md (permission errors -> 403, validation -> 400, unknown -> 500).
