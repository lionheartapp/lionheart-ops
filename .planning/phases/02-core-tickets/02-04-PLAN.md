<!-- From Plan 01 API contracts -->

GET /api/maintenance/tickets/[id] — returns full ticket with:
  submittedBy: { id, firstName, lastName, email, role: { name } }
  assignedTo: { id, firstName, lastName } | null
  building: { id, name } | null
  area: { id, name } | null
  room: { id, name, code } | null
  school: { id, name } | null
  activities: MaintenanceTicketActivity[] (filtered by isInternal based on caller permissions)

GET /api/maintenance/tickets/[id]/activities — returns filtered activity list

POST /api/maintenance/tickets/[id]/activities — body: { content, isInternal? } — adds comment

PATCH /api/maintenance/tickets/[id]/status — body: { status, holdReason?, holdNote?, completionNote?, completionPhotos?, rejectionNote?, cancellationReason? }

POST /api/maintenance/tickets/upload-url — for QA completion photo upload

<!-- From Plan 01 service layer -->
ALLOWED_TRANSITIONS map defines valid transitions from each status.
Client-side mirror needed for showing available actions.

- QA->DONE sign-off: review panel on detail page with labor/cost summary + Approve/Send Back
- Internal comments (isInternal flag) visible only to techs and Head
- Same page for all roles — submitters see fewer actions

**1a. page.tsx** — Next.js page route at `/maintenance/tickets/[id]`:
- Auth guard (redirect to /login if no token)
- DashboardLayout wrapper
- Extract `id` from params
- Render TicketDetailPage with id prop

**1b. TicketStatusTracker.tsx** — Horizontal 8-status progress bar:
- Displays all 8 statuses as steps in a horizontal tracker
- Status order: BACKLOG -> TODO -> IN_PROGRESS -> ON_HOLD -> SCHEDULED -> QA -> DONE -> CANCELLED
- Visual states for each step: completed (emerald fill + checkmark), current (emerald-500 ring + filled dot), future (gray-200 border + empty), skipped/terminal (muted)
- For CANCELLED: shown differently — if ticket is cancelled, all steps after current show "cancelled" state (red strikethrough)
- Connecting lines between steps: solid emerald for completed, dashed gray for future
- Mobile: steps compress to show current + adjacent only, with "..." for hidden
- Props: `currentStatus: MaintenanceTicketStatus`

**1c. TicketActivityFeed.tsx** — Chronological feed + comment box:
- Fetches via TanStack Query: GET /api/maintenance/tickets/[id]/activities
- Each activity entry rendered as a timeline item:
  - Left: timeline line + icon (colored circle: blue for STATUS_CHANGE, green for COMMENT, orange for ASSIGNMENT, purple for INTERNAL_NOTE)
  - Right: actor avatar (or initials) + actor name + timestamp (relative) + content
  - STATUS_CHANGE entries: "John moved ticket from IN_PROGRESS to QA" with status badges
  - COMMENT entries: actor name + comment text
  - ASSIGNMENT entries: "Jane assigned to Mike" or "Mike claimed this ticket"
  - INTERNAL_NOTE entries: shown with a "Internal" badge, only visible to privileged users (API handles filtering)
- Comment box at bottom:
  - Textarea with placeholder "Add a comment..."
  - For techs/Head: checkbox "Internal note (not visible to submitter)" — sets isInternal=true
  - "Post" button (emerald). Calls POST /api/maintenance/tickets/[id]/activities. On success, invalidates activities query + prepend optimistic entry
  - Disable when empty
- Auto-scroll to bottom of feed on new entry
- Loading: skeleton timeline items with animate-pulse
- Props: `ticketId: string, isPrivileged: boolean` (controls internal note checkbox visibility)

**1d. HoldReasonInlineForm.tsx** — Inline expansion for ON_HOLD:
- NOT a modal — expands inline below the "On Hold" action button
- Hold reason dropdown: PARTS, VENDOR, ACCESS, OTHER (from HoldReason enum)
- Optional note textarea
- "Confirm Hold" button + "Cancel" link
- On confirm: PATCH /api/maintenance/tickets/[id]/status with { status: 'ON_HOLD', holdReason, holdNote }
- Framer Motion expandCollapse for smooth expand/collapse
- Props: `ticketId: string, onComplete: () => void, onCancel: () => void`

**1e. QACompletionModal.tsx** — Modal for moving to QA:
- Uses ui-glass-overlay styling (or adapt ConfirmDialog pattern)
- Required fields: completion photo(s) — use same signed URL upload pattern from StepPhotos.tsx (at least 1 photo required), completion note (textarea, required, min 10 chars)
- "Submit for QA" button (emerald) disabled until both fields filled
- "Cancel" button
- On submit: PATCH /api/maintenance/tickets/[id]/status with { status: 'QA', completionPhotos, completionNote }
- Loading state on submit button
- Backdrop click or Escape dismisses
- Props: `ticketId: string, open: boolean, onClose: () => void, onComplete: () => void`

**1f. QAReviewPanel.tsx** — Head sign-off panel for QA tickets:
- Only rendered when: ticket.status === 'QA' AND user has MAINTENANCE_APPROVE_QA permission
- Review section showing:
  - Completion photos (thumbnails, click for full-size)
  - Completion note
  - Labor summary: total hours logged, total labor cost (from laborEntries if any — may be 0 in Phase 2 since labor tracking is Phase 4, but design for it)
  - Cost summary: total materials cost (from costEntries if any — may be 0)
- Two action buttons:
  - "Approve & Close" (emerald) — PATCH status to DONE
  - "Send Back" (red outline) — expands inline textarea for rejection note (required). "Confirm Rejection" button. PATCH status to IN_PROGRESS with rejectionNote
- Both actions fire appropriate notifications via the status transition API
- Props: `ticket: MaintenanceTicket`

**1g. TicketDetailPage.tsx** — Full two-column detail layout:
- TanStack Query: `useQuery(['maintenance-ticket', id], fetchTicketDetail)`
- Also fetch user permissions via usePermissions hook to determine: isPrivileged (has MAINTENANCE_READ_ALL or MAINTENANCE_CLAIM), isHead (has MAINTENANCE_APPROVE_QA), canChangeStatus (has MAINTENANCE_CLAIM or MAINTENANCE_ASSIGN), isSubmitter (ticket.submittedById === userId)

**Left column** (info section, ui-glass cards stacked):

Card 1 — Submitter info:
- Name (firstName + lastName), role name, email (mailto link)
- Submitted timestamp (absolute: "Mar 5, 2026 at 2:30 PM" + relative: "2 days ago")
- Availability note (if present, in amber info box)

Card 2 — Location:
- Full hierarchy: School name (if multi-campus) > Building name > Area name > Room name/code
- Google Maps link: `https://maps.google.com/?q=${encodeURIComponent(building.name)}` (new tab)
- If room has a photo (future), show it — for now, just hierarchy text

Card 3 — Issue details:
- Title (large, bold)
- Description (if present)
- Category badge + Priority badge (same styling as WorkOrdersTable)
- Photos gallery: thumbnails in grid (2 cols). Click thumbnail opens full-size in a lightbox overlay (simple: fixed position, dark backdrop, large image, close button)

Card 4 — Assignment:
- If assigned: "Assigned to [Name]" with reassign button (Head only)
- If unassigned: "Unassigned" with "Assign" button (Head only) or "Claim" button (tech if specialty matches)

**Right column** (activity + actions section):

- TicketStatusTracker at top
- Action buttons section (contextual based on current status and user role):

  - "On Hold" button -> expands HoldReasonInlineForm below
  - "Move to QA" button -> opens QACompletionModal
  - "Cancel" button (Head/Admin only) -> opens inline cancellation reason textarea
  - Submitter sees NO action buttons (only status tracker + activity + comment box)
- QAReviewPanel (if ticket.status === QA and user is Head)
- TicketActivityFeed below actions

**Layout:**
- Desktop: `grid grid-cols-1 lg:grid-cols-5 gap-6` — left column 2 cols, right column 3 cols
- Mobile: stacks vertically — info first, then status tracker + actions + activity
- Page header: "Back to Work Orders" / "Back to My Requests" link (based on referrer or role), ticket number (MT-XXXX) as title, status badge large
- Loading: skeleton matching two-column layout
- Not found: "Ticket not found" with back link

All cards use glassmorphism: ui-glass p-6 rounded-2xl. Transitions on hover for interactive elements. Framer Motion fadeInUp for card entrance stagger.
