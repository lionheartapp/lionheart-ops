**Step 1: maintenanceTicketService.ts** — Create the core service with ALL business logic:

1a. `CATEGORY_TO_SPECIALTY` constant mapping each MaintenanceCategory to its corresponding MaintenanceSpecialty (1:1: ELECTRICAL->ELECTRICAL, PLUMBING->PLUMBING, etc.)

1b. `ALLOWED_TRANSITIONS` map — a Record keyed by source status, each value a Record keyed by target status with config:
```
{ requiredPermissions: string[], requiredFields?: string[], description: string }
```
Transitions per the research:
- BACKLOG -> TODO (claim/assign), SCHEDULED (assign), CANCELLED (cancel + cancellationReason)
- TODO -> IN_PROGRESS (claim/assign), ON_HOLD (claim/assign + holdReason), CANCELLED (cancel + cancellationReason)
- IN_PROGRESS -> QA (claim/assign + completionNote + completionPhotos), ON_HOLD (claim/assign + holdReason), CANCELLED (cancel + cancellationReason)
- ON_HOLD -> IN_PROGRESS (claim/assign), CANCELLED (cancel + cancellationReason)
- QA -> DONE (approve_qa), IN_PROGRESS (approve_qa + rejectionNote), CANCELLED (cancel + cancellationReason)
- SCHEDULED -> BACKLOG (system/cron only), CANCELLED (cancel + cancellationReason)
- DONE -> (no transitions)
- CANCELLED -> (no transitions)

1c. `generateTicketNumber(orgId)` — uses rawPrisma.$transaction to atomically upsert MaintenanceCounter and increment. Returns `MT-${String(n).padStart(4, '0')}`.

1d. `createMaintenanceTicket(input, userId, orgId)` — validates with Zod (title required max 200 chars, category required, priority defaults MEDIUM, photos[] max 5 URLs, optional buildingId/areaId/roomId/scheduledDate/availabilityNote/description). Calls generateTicketNumber. Derives specialty from category. Creates ticket with status BACKLOG (or SCHEDULED if scheduledDate provided). Creates initial STATUS_CHANGE activity. Calls notification dispatch (fire-and-forget): notifyTicketSubmitted, and if priority === URGENT also notifyUrgentTicket. Returns created ticket with includes.

1e. `transitionTicketStatus(ticketId, newStatus, data, ctx)` — loads ticket, looks up transition in ALLOWED_TRANSITIONS[currentStatus][newStatus]. If not found: throw Error('INVALID_TRANSITION'). Check permissions via `canAny(ctx.userId, config.requiredPermissions)`. Validate required fields present in data. Update ticket status (and holdReason/holdNote if ON_HOLD, completionNote/completionPhotos if QA, cancellationReason if CANCELLED, rejectionNote as activity content if QA->IN_PROGRESS). Write MaintenanceTicketActivity with fromStatus/toStatus. Fire notification side effects per transition (see maintenanceNotificationService). Return updated ticket.

1f. `claimTicket(ticketId, userId, orgId)` — assertCan MAINTENANCE_CLAIM. Load user's TechnicianProfile via rawPrisma (may be null — if null, return 400 "Technician profile not found"). Load ticket. Verify ticket.specialty is in profile.specialties or ticket.specialty === 'OTHER'. If not: return 403. Set assignedToId = userId. Auto-transition BACKLOG->TODO (or if already TODO, just assign). Write ASSIGNMENT activity. Fire notifyTicketClaimed (to Head). Return updated ticket.

1g. `assignTicket(ticketId, assigneeId, userId)` — assertCan MAINTENANCE_ASSIGN. No specialty check. Update assignedToId. Auto-transition to TODO if currently BACKLOG. Write ASSIGNMENT activity. Fire notifyTicketAssigned (to assignee). Return updated ticket.

1h. `getTicketDetail(ticketId, userId)` — findUnique with includes: submittedBy (id, firstName, lastName, email, role.name), assignedTo (id, firstName, lastName), building (id, name), area (id, name), room (id, name, code), school (id, name), activities (with actor and assignedTo, ordered by createdAt asc). Filter activities: if user does NOT have MAINTENANCE_READ_ALL or MAINTENANCE_CLAIM permission, exclude isInternal=true entries. Return ticket + filtered activities.

1i. `listTickets(filters, ctx)` — role-scoped: if has MAINTENANCE_READ_ALL, return all tickets with filters. If has MAINTENANCE_CLAIM, return unassigned + own assigned tickets; also load TechnicianProfile and add `matchesSpecialty: boolean` to each result. Otherwise return only submittedById === ctx.userId. Support filters: status, priority, category, schoolId (campus), assignedToId, search (title ILIKE), unassigned (assignedToId is null), excludeStatus (for hiding SCHEDULED from main list). Default sort: priority desc (URGENT first), then createdAt asc. Include submittedBy, assignedTo, building, room, school.

**Step 3: API Routes** — follow established pattern (getOrgIdFromRequest -> getUserContext -> assertCan -> runWithOrgContext -> prisma):

3a. `POST /api/maintenance/tickets` — assertCan MAINTENANCE_SUBMIT. Parse body with Zod. Call createMaintenanceTicket. Return 201 with ok(ticket).

3b. `GET /api/maintenance/tickets` — assertCan MAINTENANCE_READ_OWN (minimum). Parse query params for filters. Call listTickets. Return ok(tickets).

3c. `GET /api/maintenance/tickets/[id]` — assertCan MAINTENANCE_READ_OWN. Call getTicketDetail. If user only has READ_OWN and ticket.submittedById !== userId, return 403. Return ok(ticket).

3d. `PATCH /api/maintenance/tickets/[id]` — assertCan MAINTENANCE_UPDATE_OWN (for metadata updates like adding comment). Support adding photos (append to photos array). Return ok(updated).

3e. `PATCH /api/maintenance/tickets/[id]/status` — call transitionTicketStatus. Catch 'INVALID_TRANSITION' -> 400. Return ok(updated).

3f. `GET /api/maintenance/tickets/[id]/activities` — assertCan MAINTENANCE_READ_OWN. Filter isInternal based on permissions. Return ok(activities).

3g. `POST /api/maintenance/tickets/[id]/activities` — assertCan MAINTENANCE_READ_OWN (anyone can comment). Parse body with Zod (content required, isInternal boolean default false). Create MaintenanceTicketActivity with type COMMENT (or INTERNAL_NOTE if isInternal and user has MAINTENANCE_READ_ALL or MAINTENANCE_CLAIM). Return 201 with ok(activity).

3h. `POST /api/maintenance/tickets/[id]/claim` — call claimTicket. Return ok(updated).

3i. `POST /api/maintenance/tickets/upload-url` — assertCan MAINTENANCE_SUBMIT. Parse body (fileName, contentType). Generate signed upload URL via Supabase Storage createSignedUploadUrl for bucket 'maintenance-photos' with path `${orgId}/${Date.now()}-${fileName}`. Return ok({ signedUrl, publicUrl, path }).

3j. `POST /api/maintenance/tickets/ai-suggest-category` — assertCan MAINTENANCE_SUBMIT. Parse body (imageBase64 or imageUrl). Call Gemini gemini-2.0-flash with image + classification prompt (see Pattern 5 in research). Return ok({ suggestedCategory }). Wrap in try/catch — on failure return ok({ suggestedCategory: null }) so it never blocks. If GEMINI_API_KEY not set, return ok({ suggestedCategory: null }).

3k. `POST /api/maintenance/tickets/ai-detect-multi-issue` — assertCan MAINTENANCE_SUBMIT. Parse body (title, description, category). Send to Gemini with prompt asking if this describes multiple distinct issues. Return ok({ hasMultipleIssues, secondIssue?: { title, suggestedCategory } }). Gracefully degrade on failure.

3l. `GET /api/maintenance/dashboard` — assertCan MAINTENANCE_READ_ALL. Aggregate: count tickets by status, count by priority, count by category, count unassigned, count overdue (BACKLOG older than 48h), average resolution time (DONE tickets: updatedAt - createdAt). Accept optional schoolId filter. Return ok(stats).

3m. `GET /api/cron/maintenance-tasks` — verify CRON_SECRET from Authorization header. Use rawPrisma (no org context — iterates all orgs). Task 1: find SCHEDULED tickets where scheduledDate <= now(), transition each to BACKLOG, write activity, fire notifyStatusChange. Task 2: find BACKLOG tickets with no assignedToId and createdAt < 48h ago and staleAlertSent === false, send notifyStaleTicket for each, set staleAlertSent = true. Return ok({ released: N, alerted: M }).
