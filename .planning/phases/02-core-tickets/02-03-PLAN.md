**1a. WorkOrdersFilters.tsx** — Filter bar component:
- Horizontal bar with: status dropdown (multi-select, all 8 statuses), priority dropdown (multi-select), category dropdown (multi-select, 8 categories), campus dropdown (single-select from user's schools), technician dropdown (single-select, populated from org technicians), keyword search input (text, debounced 300ms), unassigned toggle (checkbox)
- "Clear filters" button that resets all to defaults
- All filter state managed via `useState` in parent (WorkOrdersView), passed as props + onChange callbacks
- Compact design: use flex-wrap so filters stack on mobile. Each filter uses small dropdown/input styling
- Inputs use established input classes: `w-full px-3 py-2 border border-gray-200 rounded-lg text-sm bg-white`

**1b. WorkOrdersTable.tsx** — Main ticket table:
- Uses `ui-glass-table` wrapper
- Columns: ticket # (sortable), title, status (badge), priority (badge, sortable), category (tag), location (Building > Room), assigned tech (name or "Unassigned" in muted), age (relative time — "2d", "5h", sortable)
- Default sort: priority desc (URGENT=4, HIGH=3, MEDIUM=2, LOW=1) then age asc (oldest first) per user decision
- Column headers clickable for sort direction toggle (show sort arrow icons)
- Each row: hover state with `bg-gray-50/50` transition, cursor-pointer (row click navigates to `/maintenance/tickets/[id]`)
- Inline quick action: "..." button at end of each row, opens a dropdown menu (ui-glass-dropdown) with:
  - "Claim" — only shown if: (a) ticket is unassigned, (b) user has MAINTENANCE_CLAIM permission, (c) ticket.matchesSpecialty is true. Call POST /claim with optimistic TanStack Query update: immediately mark as assigned in UI, rollback on error
  - "Assign" — only shown if user has MAINTENANCE_ASSIGN permission. Opens a small popover with technician select dropdown + "Assign" button
  - "Change Status" — only shown if user has MAINTENANCE_CLAIM or MAINTENANCE_ASSIGN. Opens popover with dropdown of valid next statuses (client-side ALLOWED_TRANSITIONS mirror) + required field inputs (holdReason, completionNote, etc.) + "Update" button
- Props: tickets array, isLoading, specialties (for highlighting), onClaim, onAssign, onStatusChange callbacks

**1c. Specialty highlighting for technicians:**
- If the API returns `matchesSpecialty` field on tickets (technician view): rows where matchesSpecialty === false get `opacity-50` and grayed-out text
- "Show all" toggle in WorkOrdersView header: when OFF (default for techs), client-side filter to only show matchesSpecialty === true tickets. When ON, show all with grayed highlighting
- For Head/Admin users: no specialty filtering, all tickets shown at full opacity

**1d. WorkOrdersView.tsx** — Container component:
- Header row: "Work Orders" title + action buttons area
- For techs: "Show all" toggle switch
- WorkOrdersFilters rendered below header
- TanStack Query: `useQuery(['maintenance-tickets', filters], fetchTickets)` — passes all filter params to GET /api/maintenance/tickets (excludeStatus=SCHEDULED for main table, excludeStatus=DONE,CANCELLED optional or as tab)
- Pass filtered data to WorkOrdersTable
- Below main table: collapsible "Scheduled Tickets" section using Framer Motion expandCollapse variant:
  - Section header: "Scheduled (N)" with chevron toggle
  - Sorted by scheduledDate (soonest first)
  - Same table columns but with scheduledDate shown instead of age
  - Collapsed by default
- Optimistic mutation for claim: `useMutation` with `onMutate` to update cache immediately, `onError` to rollback, `onSettled` to invalidate queries per Pattern 7 in research
- Loading state: skeleton table with animate-pulse rows
- Empty state: "No work orders match your filters" with suggestion to clear filters

**Table must be responsive:** On mobile, collapse to a card-like list view instead of a full table. Show ticket #, title, status badge, priority badge on each card-row. Hide less critical columns (category, location, age) behind a "..." expand.
